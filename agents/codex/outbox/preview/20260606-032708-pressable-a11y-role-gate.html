P1first-run and core workflow issue

Backdrop wrappers are not automatically bugs. The blocker is visible controls that act as buttons, tabs, filters, or choices without role/state parity.

P1 Findings

Sign-in

src/app/(auth)/sign-in.tsx:224, :241, :253, :265, :277 raw buttons lack explicit role/state. This is the first screen.

Inbox

src/app/inbox.tsx:96, :129, :149, :168, :189, :197, :523, :540 are row/action Pressables without role/state; generateBtn is still only 30px min-height.

Capture

Track chips, mode tabs, show/hide controls, Advisor opt-in, and removable tag chips lack button/tab/checkbox semantics and selected/checked state.

Wiki, Research, Assessments

Tag filters, DOI links, row expanders, Big Five/ECR Likert options, and intro checkbox rows need role/state semantics, not just active color.

Consent Dialog

ConsentDialog.tsx wraps the card inside a backdrop Pressable that declines. Runtime proof or restructuring is needed so inner taps cannot accidentally dismiss consent.

Recommended Primitives

ActionChip
44px min, role button, disabled/busy state.
SelectableChip
selected state for filters, tracks, and mode choices.
TextLinkButton
role link/button, explicit label, 44px target.
CheckboxRow
role checkbox, checked state, label.
