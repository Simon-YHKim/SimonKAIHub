# ai-model-selector — 사용자 제품의 모델·프로바이더 결정

> 사용자가 만드는 **AI 제품/기능**에서 어떤 LLM·VLM·음성 모델을, 어느 프로바이더로, API 로 부를지 self-host 할지 결정한다.
> ⚠️ 혼동 금지: `model-router` 는 **이 스택(Claude Code)** 의 내부 task→model 라우팅이다. 이 스킬은 **사용자 앱 코드가 호출할** 모델을 고른다.

## 0. 운영 원칙

- 모델명은 환각 금지. 최신 frontier 만 참조: **Claude Opus 4.8 / Sonnet 4.6 / Haiku 4.5 / Fable 5**, **Gemini 2.x**, 오픈모델(Llama·Qwen·DeepSeek·Mistral 계열). 옛 모델명 하드코딩 금지.
- 가격·컨텍스트·모달리티 수치는 **추론으로 단정하지 말고** WebSearch/WebFetch 로 vendor 공식 페이지를 확인한 뒤 제시. 확인 못 하면 "확인 필요"로 표기.
- Anthropic 모델 세부(가격/ID/파라미터)는 `claude-api` 스킬 참조가 1차 출처.
- 최종 결정 전 비용 한도·프라이버시 제약은 **사용자에게 묻는다**(AskUserQuestion). 단정 후 진행 금지.

## 1. 5단계 결정 깔때기

순서대로 좁힌다. 앞 단계가 뒤 단계를 제약한다.

| 단계 | 질문 | 산출 |
|---|---|---|
| 1. 작업 적합성 | 무슨 작업인가? (추론·코딩·요약·분류·비전·음성·임베딩) | 후보 모델군 |
| 2. 운영 제약 | 비용/레이턴시/컨텍스트 길이/필요 모달리티 한도는? | 후보 압축 |
| 3. 데이터 거버넌스 | 개인정보·리전·온프레 요구? 데이터 학습 사용 금지? | API 가능 여부 |
| 4. 배포 형태 | API(managed) vs 오픈모델 self-host vs 온디바이스 | 호스팅 결정 |
| 5. 견고성 | 라우팅·폴백·캐싱·레이트리밋 전략 | 운영 설계 |

### 1단계 — 작업 적합성

| 작업 유형 | 신호 | 1차 후보 | 2차 후보 | 저비용/대량 |
|---|---|---|---|---|
| 복잡 추론·에이전트 | multi-step, tool use, 기획·계획 | Claude Opus 4.8 | Gemini 2.x Pro | Claude Sonnet 4.6 |
| 코딩 생성·수정 | 코드 작성·리팩토링·PR | Claude Sonnet 4.6 | Claude Opus 4.8 | 오픈 코드모델(self-host) |
| 일반 대화·작문 | 챗봇, 카피, 문서 | Claude Sonnet 4.6 | Gemini 2.x Flash | Claude Haiku 4.5 |
| 대량·경량 분류/태깅 | bulk, 라벨링, 라우팅 전처리 | Claude Haiku 4.5 | Gemini 2.x Flash | 오픈 소형모델 |
| 롱컨텍스트 분석 | 대용량 문서·코드베이스 | Gemini 2.x Pro(롱컨텍스트) | Claude Opus 4.8 [1M] | — |
| 비전·멀티모달 | 이미지·도표·스크린샷·OCR | Gemini 2.x Pro | Claude Opus 4.8 | Gemini 2.x Flash |
| 이미지 생성 | 일러스트·에셋 | Fable 5 | Gemini 2.x(이미지) | 오픈 디퓨전(self-host) |
| 음성(STT/TTS·실시간) | 통화·받아쓰기·보이스봇 | Gemini 2.x(라이브/오디오) | 전용 STT/TTS 프로바이더 | 온디바이스 STT |
| 임베딩·검색 | RAG, 시맨틱 검색 | 전용 임베딩 모델 | 오픈 임베딩(self-host) | — |

> 한국어 비중이 높은 제품: 1차 후보 그대로 두되, 실제 프롬프트 샘플로 한국어 품질을 **직접 평가**(아래 5절 eval)한 뒤 확정. "한국어니까 X" 같은 선입견으로 결정하지 않는다.

### 2단계 — 운영 제약 (비용·레이턴시·컨텍스트·모달리티)

- **비용**: 토큰당 입력/출력 단가 × 예상 호출량 → 월 추정. 입력≫출력이면 프롬프트 캐싱 효과 큼.
- **레이턴시**: 사용자 대면 동기 응답이면 Flash/Haiku 급 + 스트리밍. 배치/백그라운드면 대형 모델 허용.
- **컨텍스트**: 한 호출에 넣을 최대 토큰. 롱컨텍스트가 상시 필요한지, 가끔인지로 모델/요금제 분기.
- **모달리티**: 텍스트 전용인지 비전/오디오 동반인지. 동반이면 멀티모달 네이티브 모델 우선(파이프라인 단순화).
- 가격이 결정 요소면 `paid-api-guard`(Stack) 관점 연계: 무료티어 한도, 예산 가드, 호출 상한.

### 3단계 — 데이터 거버넌스

| 요구 | 함의 |
|---|---|
| 개인정보(PII)·민감정보 다룸 | DPA·데이터 학습 비사용(no-train) 옵션 확인. 미가공 PII 전송 최소화 |
| 리전 제약(국내 보관) | vendor 리전 엔드포인트 유무. 없으면 self-host/온디바이스 검토 |
| 망분리·온프레 의무 | managed API 불가 → 오픈모델 self-host 또는 온디바이스 강제 |
| 규제 산업(의료·금융·공공) | 감사 로그·계약상 보증 필요. 2nd-Brain 의 `ai_audit_log`(C3) 같은 호출 로깅 패턴 권장 |

→ 거버넌스가 managed API 를 막으면 **4단계는 self-host/온디바이스로 직행**.

### 4단계 — API vs 오픈모델 self-host vs 온디바이스

| 선택지 | 적합 | 장점 | 비용·운영 부담 |
|---|---|---|---|
| Managed API | 대부분의 제품, 빠른 출시 | 최신 모델·확장성·운영 0 | 토큰 단가, 데이터 외부 전송 |
| 오픈모델 self-host | 프라이버시·대량 정형 트래픽·리전 의무 | 데이터 통제, 대량시 단가↓ | GPU·MLOps·서빙(vLLM/TGI) 운영 |
| 온디바이스 | 오프라인·초저지연·완전 프라이버시 | 네트워크 0·비용 0·프라이버시 최상 | 모델 크기·기기 성능·품질 한계 |

판단 규칙:
- 트래픽 적고 품질 최우선 → **Managed API**(거의 항상 시작점).
- 월 호출량이 self-host 손익분기를 넘고 운영 역량 있음 → **self-host** 검토.
- 망분리/오프라인/리전 강제 → self-host 또는 온디바이스 외 선택지 없음.
- 모바일·엣지에서 즉답 + 프라이버시 → **온디바이스 소형모델** + 어려운 요청만 API 로 escalate(하이브리드).

> 2nd-Brain 처럼 모바일(RN/Expo) 제품은 기본 managed API, 보안 분류·민감 처리만 온디바이스/엣지로 분리하는 하이브리드가 흔하다. RN/Expo 결정은 `app-platform-selector` 와 연계.

### 5단계 — 라우팅·폴백·캐싱 (견고성)

- **라우팅**: 쉬운 요청은 소형(Haiku/Flash), 어려운 요청만 대형(Opus/Pro). 분류기로 1차 라우팅 후 escalate.
- **폴백 체인**: Primary → Secondary(타 프로바이더 가능) → 저비용. 한 vendor 장애·레이트리밋 시 자동 전환.
- **캐싱**: ① 프롬프트(시스템·few-shot) 캐싱으로 입력 토큰비 절감, ② 동일 입력 응답 캐시로 중복 호출 제거.
- **레이트리밋·재시도**: 지수 백오프, 큐잉, 예산 상한(일/월) 가드.
- 폴백을 타 프로바이더로 둘 땐 응답 스키마·tool-call 포맷 차이를 어댑터로 흡수.

## 2. 사용 흐름

### 패턴 A — 단일 기능 모델 추천
1. 사용자 요청에서 작업 유형 분류(1단계).
2. 비용/레이턴시/모달리티·거버넌스가 불명확하면 **AskUserQuestion** 으로 1~3개 핵심만 질문.
3. vendor 공식 가격/스펙 WebFetch 확인.
4. 1차/2차/저비용 후보를 표(아래 4절)로 출력 + 한 줄 사유.

### 패턴 B — 제품 전체 모델 전략
- 기능별로 패턴 A 를 반복 → `templates/model-decision.json` 채워 라우팅 맵 산출.
- 폴백·캐싱·예산 가드 설계(5단계) 동봉.

### 패턴 C — 마이그레이션 검토
- 기존 모델 대비 신규 후보를 동일 eval 셋(아래)으로 비교 → 품질 동등/우위 + 비용 변화 표로 결정.

## 3. 의사결정 전 확인 질문 (AskUserQuestion)

다음이 불명확하면 진행 전에 묻는다. 한 번에 핵심 1~3개만.

- 월 예상 호출량 / 허용 월예산 (비용 가드 산정)
- 사용자 대면 동기 응답인가, 배치인가 (레이턴시)
- PII·민감정보·리전·망분리 제약 여부 (거버넌스 → self-host 분기)
- 비전/오디오 등 멀티모달 필요 여부
- 단일 프로바이더 의존 허용 여부 (폴백 설계)

## 4. 출력 형식

### Markdown (기본)
```
작업 유형: <분류>
| 순위 | 모델 | API ID / 호스팅 | 사유 | 가격(확인필요 시 표기) |
|---|---|---|---|---|
| 1차 | Claude Opus 4.8 | claude-opus-4-8 (Anthropic API) | 복잡 추론·에이전트 최상 | $.../$... |
| 2차 | Gemini 2.x Pro | gemini-2.x-pro (Vertex/AI Studio) | 롱컨텍스트·비전 | $.../$... |
| 저비용 | Claude Haiku 4.5 | claude-haiku-4-5 | 1차의 일부 비용, 라우팅 1차 처리 | $.../$... |

거버넌스: <managed API 가능 / self-host 권장 / 온디바이스>
폴백 체인: 1차 → 2차 → 저비용
캐싱: <프롬프트 캐싱 / 응답 캐시 적용 지점>
```

### JSON (제품 라우팅 맵 — `templates/model-decision.json`)
기능별 1차/2차/저비용 + 폴백·캐싱·예산을 한 파일로. 앱의 모델 라우팅 레이어가 consume.

## 5. 모델 검증(eval) — 단정 대신 측정

추천을 확정하기 전, 사용자 제품의 **실제 프롬프트 5~20개**로 후보들을 비교한다.
- 동일 입력 → 후보별 출력 품질(정확도/형식 준수/한국어 자연스러움) + 토큰비 + 레이턴시 기록.
- 표로 비교 후 1차 확정. "벤치마크 1등이니까"가 아니라 **이 제품 작업에서의 성능**으로 결정.
- 마이그레이션(패턴 C)도 같은 eval 셋 재사용해 회귀 방지.

## 6. 안전 가드

- **모델명 환각 금지**: vendor 공식 페이지에서 확인된 모델/ID 만 제시. 미확인 마케팅 명칭 추천 X.
- **가격 단정 금지**: 가격이 결정 요소면 반드시 공식 페이지 확인. 옛 단가 기억으로 답하지 않는다.
- **비용 폭증 방지**: 대형 모델 × 고빈도 호출 추천 시 월 추정치 명시 + `paid-api-guard` 권장.
- **프라이버시 우선**: PII/민감/리전 제약이 잡히면 managed API 를 기본값으로 밀지 말고 self-host/온디바이스를 동등 비교.
- **단일 의존 경고**: 폴백 없는 단일 프로바이더 설계는 장애·레이트리밋 리스크 고지.

## 7. 관련 스킬·자산

- `claude-api` — Anthropic 모델 ID·가격·파라미터·캐싱·tool use 의 1차 출처.
- `model-router` — **이 스택 내부** task→model 라우팅(혼동 금지).
- `paid-api-guard` (Stack) — 무료티어·예산 가드·호출 상한.
- `app-platform-selector` — 하이브리드/네이티브/온디바이스 배포 결정 연계.
- `db-selector` — 임베딩·벡터 저장소(RAG) 결정 시 함께.
- 템플릿: `templates/model-decision.json` — 제품 모델 라우팅 맵.

## 8. Roadmap

- v1.0 (현재): 5단계 깔때기 + 작업유형 매트릭스 + Markdown/JSON 출력 + eval 가이드.
- v1.1: vendor 가격/스펙 실시간 fetch 캐시(WebFetch) + 월비용 자동 추정기.
- v1.2: 제품 라우팅 맵 → 앱 코드 어댑터 스캐폴드(프로바이더 추상화) 생성.
