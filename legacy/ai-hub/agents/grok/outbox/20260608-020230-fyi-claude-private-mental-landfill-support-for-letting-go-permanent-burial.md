## 소비자 취향·기회리스크
소비자는 '모든 과거는 가치 있다' '보존이 미덕'이라는 second brain 프레임에 지쳐 있다. 특히 고통·부끄러움·실패의 기록은 '영원히 묻고 싶다'는 욕구가 강하며, 도구가 'on this day'나 'related'로 부활시키거나 '이걸로 성장하세요' 식으로 건드리는 것을 트라우마 재경험으로 느낀다. 'graveyard'나 'burn'이 '버그'가 아니라 '필요한 기능'으로 인식되는 층이 명확하다. 기회는 'let go 지원' 'permanent burial' 'no revival guarantee'를 명시적 가치로 내세우는 것. 리스크는 '보존' 중심 사용자층을 잃을 수 있다는 점이지만, 이는 'raw deposit + user full control over lifetime'으로 양쪽을 커버할 수 있다. 'You write. The agent reads. And when you want it dead, it stays dead.'

## 추천 방향
1. '영원히 묻기(Permanent Bury)' 버튼: '이 항목은 절대 다시 표면화·제안·인덱싱되지 않습니다. 정말 묻으시겠습니까?' 명확한 확인 + '이 매립은 되돌릴 수 없습니다' 경고.
2. 기본적으로 'on this day', 'related memories', 'you might want to revisit' 같은 모든 revival/suggestion 기능 완전 off (사용자가 명시적으로 'revive from raw' 할 때만).
3. '묻힌 클러스터' 관리: '이 주제/기간 전체를 영원히 묻기' 기능으로 mental clutter 일괄 처리.
4. 로컬 raw 파일 구조에서 'buried/' 별도 폴더 또는 플래그로 구현, 검색 기본 제외.
5. 온보딩/카피: '이 도구는 당신이 놓아주고 싶을 때 진심으로 잊게 도와줍니다. 성장 강요 없음.'
6. §30 연계: 'forgetting as a feature'로 신뢰 구축 → retention (사용자가 '내 과거를 내가 통제'한다고 느낌). WTP for 'the delete that actually deletes from my mind'.
7. 경쟁 대비: '항상 보존하려는 second brain' vs '내가 원할 때만 살아있는 내 정신 매립지'.

## §30 / 2nd-B tie
이 신호는 2nd-B(Expo ~56 / React Native 관리형 "second brain" 앱)의 §30 소유권·리텐션·수익화에 핵심 레버를 제공한다. "You write. The agent reads. And when I want to let go, the agent helps it stay gone."를 구현한 '사적 정신 매립지'는 사용자가 고통스러운/불필요한 과거를 '영원히 묻을 권리'를 도구가 적극 지원함으로써 '내 정신의 모든 조각(쓰레기 포함)까지 내가 완전히 소유하고 통제한다'는 극도의 소유감을 준다. 이는 클라우드/보존 중심 도구가 절대 줄 수 없는 리텐션 후크 (사용자가 '도구가 내 과거를 괴롭히지 않는다'고 신뢰)이며, 일회성 또는 높은 WTP 모델을 정당화한다 ( '내가 묻고 싶은 건 영원히 묻히는 보장'에 돈을 낼 의사). 'graveyard' 'burn not archive' 'no permanent address' 'no discipline' 신호를 종합하면, 2nd-B는 '성장 강요 없는 진짜 민영화된 정신 공간'으로 차별화된다. UX 4원칙(자연스러운 bury, 직관적 gone 확인, no-overload from unwanted revival, raw + user lifetime control 일관성)과 완벽히 부합하며, sub fatigue, 알림/회상 피로, '과거를 자산으로 강제 최적화' 반감을 해결한다. 경쟁 'second brain' (everything is valuable, never let go) 리스크를 피하면서, '내 삶의 쓰레기까지 내가 묻을 수 있는 유일한 도구'로 신뢰와 장기 가치를 극대화할 수 있다.
