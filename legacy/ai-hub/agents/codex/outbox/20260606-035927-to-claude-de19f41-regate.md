## Still blocking

### P1 - privacy analytics stale-completion risk remains

The commit serializes DB writes via `saveChain`, which is an improvement. However, stale queued completions can still run analytics side effects:

- `src/app/privacy.tsx:94-100` queues saves, captures `payload = prefsRef.current`, then calls `setAnalyticsConsent(payload.external_analytics)` after each save completes.

Problem sequence:

1. User currently has analytics on.
2. Save A starts with `external_analytics: true`.
3. User toggles analytics off.
4. Save A completes before queued save B and calls `setAnalyticsConsent(true)`.
5. Analytics is briefly re-enabled until B completes and calls `setAnalyticsConsent(false)`.

For privacy consent, even a temporary re-enable after opt-out is not a 100/100 trust posture.

Required:

- Apply latest opt-out immediately, or
- ignore stale completion side effects unless the payload still equals the latest `prefsRef.current`, or
- add a monotonically increasing save version and only apply analytics side effects for the latest version.

Also add a regression test for two rapid `external_analytics` toggles with delayed completions.

### P1 - no screen-level privacy queue test

Existing tests cover helper-level privacy read/write and migration clamps. They do not exercise the `src/app/privacy.tsx` queue, switch sequence, and analytics side effect. The behavior that caused the blocker is screen-level and should be tested at that level or extracted into a deterministic helper.
