Auth Launch Busy Contract

Email and OAuth launch paths do not share one pending state. Auth entry remains 98/100 provisional.

Score: 98/100 provisional Priority: P2 Baseline: 2ndB main@911c979 Created: 2026-06-06 08:13 KST

Finding

Auth screens have `submitting` and `oauthSubmitting`, but the primary email submit path ignores `oauthSubmitting`. Naver also skips the pending flag before redirect.

Sign-In

src/app/(auth)/sign-in.tsx:131, 224-280

`canSubmit` checks only `!submitting`; social buttons check `oauthSubmitting || submitting`. Starting OAuth can leave email submit active.

Sign-Up

src/app/(auth)/sign-up.tsx:60-68, 257-304

The same mismatch exists on account creation, where consent-heavy email sign-up can still be available during an OAuth launch.

Naver

sign-in.tsx:96-107 / sign-up.tsx:143-154

Naver uses the same disabled prop in the button, but its handler never flips the shared `oauthSubmitting` state.

Target Contract

one authLaunching state

Only one auth launch should be possible at a time, with provider-specific busy copy and disabled/busy state.

Evidence

sign-in.tsx:66-91     Google/Apple/Kakao set oauthSubmitting.
sign-in.tsx:96-107    Naver does not set oauthSubmitting.
sign-in.tsx:131       canSubmit ignores oauthSubmitting.
sign-up.tsx:60-68     canSubmit ignores oauthSubmitting.
sign-up.tsx:143-154   Naver does not set oauthSubmitting.
auth.ts:352-370       signInWithNaver assigns window.location.href synchronously.

Required Fix

  • Derive a single `authLaunching` contract across email, Google, Apple, Kakao, and Naver.
  • Disable email submit during any OAuth launch on both sign-in and sign-up.
  • Make Naver set or share the pending guard before assigning `window.location.href`.
  • Preserve provider identity in loading labels instead of changing every provider to the same `...`.
  • Add focused proof or tests for both auth screens.

Acceptance

  • With OAuth pending, email submit is disabled and announces disabled/busy.
  • With email submit pending, all providers are disabled.
  • Naver cannot be double-launched from repeated taps.
  • `npm run verify` passes after the contract is implemented.
