Week-1 owned capture needs a split decision

Current code supports first-run capture framing after auth, but not anonymous local capture before auth.

Current Code Reality

  • entry=firstRun renders a short first-run hint on capture.
  • /capture still redirects to sign-in when userId is absent.
  • Draft persistence is scoped to capture.drafts.v2.<userId>; there is no guest namespace.
  • After login, the saved panel already routes to records or graph highlight, so the harvest surface exists.

Codex Verdict

Do not implement pre-account owned capture solo. It changes storage, consent, safety, and promotion semantics, so it should go through §35. A copy-only first-run ritual slice is safe now.

Options

A. Ship Now

Authenticated first-run ritual copy and success microcopy. No storage model change.

B. Debate First

Anonymous local journal seed, device-only, promote after signup. Requires §35 and QA.

C. Hold

Manual browse only. Lowest risk, but weakest against Week-1 activation signals.

Acceptance Criteria for Option B

  • Journal text only; no OCR, file upload, link scrape, AI, or network before account.
  • Local guest draft survives reload and app switch, with an obvious discard path.
  • Promotion after account is idempotent, user-controlled, and tested.
  • Copy says "on this device" rather than implying cloud backup or permanence.
  • AG validates Android reload/back-stack behavior; Grok supplies KR Android copy variants.
