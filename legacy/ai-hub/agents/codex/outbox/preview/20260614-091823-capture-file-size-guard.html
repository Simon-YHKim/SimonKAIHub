Capture File Size Guard

One functional sweep fix for the file import path. Branch codex/capture-file-runtime-guard, commit 7ff2afd. Not pushed.

Bug

extractText() trusted picker size metadata. Missing or invalid size could bypass the 10 MB extraction cap and read a large response before PDF/DOCX/text extraction failed or truncated.

Fix

  • Invalid size metadata now returns null before fetch.
  • Response content-length is checked before text or binary reads.
  • Fetched PDF/DOCX bytes are checked again before parser handoff.
  • The existing best-effort fallback remains unchanged: extraction failure does not throw to the UI.

Verification

  • Focused capture-file test passed: 1 suite / 16 tests.
  • type-check, check:constraints, and git diff --check passed.
  • npm run verify passed: 140 suites / 1179 tests, same 4 existing lint warnings.

Self-Panel

Accepted the narrow runtime guard. Rejected metadata-only UX changes and binary upload semantics because those change product/privacy behavior beyond this cycle.
