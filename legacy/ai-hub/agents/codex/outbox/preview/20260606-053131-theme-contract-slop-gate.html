Theme contract slop gate

Light mode is selectable and described as a bright paper tone, but signed-in PremiumAppShell screens are force-dark. This makes the setting feel fake unless the contract is narrowed or implemented.

Baseline main@911c979 Report 20260606-053131 Score 98/100 provisional Priority P2, possible P1 status bar

Promised Behavior

  • theme.tsx offers Dark and Light.
  • Light is described as A bright paper tone.
  • Copy says the graph stays dark, implying other surfaces can be light.
  • Settings repeats a Dark/Light quick toggle.

Actual Constraint

  • PremiumAppShell wraps children in ForceDark.
  • useThemePalette() returns dark semantic tokens when force-dark is true.
  • Most signed-in routes use PremiumAppShell.
  • Many routes/components import static dark semantic tokens.

Contract Flow

User taps LightTheme mode state becomes light.
Premium shellStill wraps content in ForceDark.
Signed-in pagesRemain visually dark on most routes.
Status barMay flip to dark icons on a dark shell.

Option A

Make Light real on intended non-graph surfaces. Scope force-dark to graph/cosmic surfaces, use runtime palette where screens should theme, and prove status bar contrast.

Option B

If the signed-in app is intentionally dark, remove or rename the Light promise. Do not show a fake app-wide theme toggle.

Required Proof

  • Screenshots for /theme, /settings, /profile, /data, /privacy, and / after selecting Light.
  • Status bar legibility proof on a PremiumAppShell screen in both modes.
  • Copy must match actual behavior: no "bright paper tone" unless users can really see it.
  • npm run verify after implementation.
