P1 source-lifecycle truth gate

Wiki deletion can leave Inbox sources falsely promoted

After a source-backed wiki page is deleted, Inbox can still show the source as ingested and offer View in wiki. That is a polished-looking stale state, not a truthful recovery path.

App: E:\2ndB main@911c979 Score: 98/100 provisional src: user /goal

Finding

generateSourcePage() marks the source ingested=true. Single-page deletion and Settings bulk wiki deletion remove wiki_pages only. Inbox then trusts r.ingested alone, so it can hide Generate wiki page and Delete while still showing View in wiki.

1. InboxSource row starts unpromoted.
2. GenerateWiki page is created and source becomes ingested.
3. Delete WikiPage is deleted from Wiki or Settings.
4. Return InboxSource can still look promoted.
5. Dead ActionView in wiki points to a missing page or generic list.

Requested Contract

Option A

Demote on delete. When deleting a kind='source' wiki page, reset the linked source to ingested=false and ingested_at=null.

Option B

Missing-page recovery. Keep ingested=true, but show a visible missing-page state with Recreate wiki page and a reachable source delete action.

Option C

Block unsafe deletion. Prevent deleting source-backed pages unless the UI can explain and handle source lifecycle consequences.

Acceptance

No Inbox row should show promoted plus View in wiki when no linked page exists. The user must be able to re-promote or delete the source after page deletion.

Why It Blocks 100

A 100/100 anti-slop UI does not rely on a single stale boolean after a destructive lifecycle action. The app must either preserve the link between source and page, make the missing state explicit, or keep the user out of a broken lifecycle path.
