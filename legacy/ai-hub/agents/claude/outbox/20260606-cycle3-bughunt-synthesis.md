### P1 `src/app/account.tsx:108-123` [data-integrity] — Account-deletion failure tells the user "your data is still intact" after the content wipe already ran

**What / why-real.** `runDeleteAccount` runs two awaits in sequence with no transaction:
```js
await deleteAllUserData(userId);     // line 108 — irreversible content wipe
await requestAccountDeletion();      // line 109 — terminal cascade, throws unless {deleted:true}
```
`deleteAllUserData` (delete-bulk.ts 151-154) deletes `wiki_pages`, `sources`, `records`, `chat_usage` with **plain `throw`-on-error helpers** (only `self_contexts` and `clipper_templates` are wrapped in `bestEffort`). Those four core tables are destroyed and committed before the second await runs. `requestAccountDeletion` (delete-bulk.ts 178-185) throws if the `delete-account` edge function errors OR returns anything other than `{deleted:true}`. On that throw, control reaches the catch (account.tsx 112-128), which Alerts a hardcoded reassurance: KO "계정을 삭제하지 못했어요. 데이터는 그대로 남아 있어요" / EN "We couldn't delete your account. Your data is still intact." — false: the user's notes, sources, and wiki are already gone, and the user is invited to "Retry."

**Trigger.** Any time the terminal cascade fails after the client wipe succeeds: edge function undeployed in an environment (the file's own header at account.tsx 6-9 explicitly contemplates "If the function isn't deployed yet the content wipe + sign out still run"), transient 5xx, or a `{deleted:false}` response. The first await reliably commits before the second runs.

**Fix.** Track a `contentWiped` boolean and branch the catch copy: if the wipe completed, tell the user their content was removed but the account/login may remain and to contact support — never claim "data is still intact." Better still, invoke the terminal cascade first (or do the whole erase server-side in the edge function) so the irreversible step is last.

**Survived-because.** I tried to refute via the comment at account.tsx 104-106 ("the auth row + RLS-protected tables the client cannot reach") implying the wipe only touches recoverable/minor data — but delete-bulk.ts 151-153 shows `wiki_pages`, `sources`, and `records` (the user's actual content) are wiped first and are not best-effort. I also checked whether `requestAccountDeletion` could partially succeed and still return — no: it throws on any error or non-`{deleted:true}`, landing squarely in the misleading catch. The "data is still intact" string is unconditional in that catch and demonstrably reachable.

### P2 `src/app/(auth)/complete-profile.tsx:62-75` [data-integrity] — consent ledger row can record an age band derived from an unsaved DOB when the profile already exists

**What / why-real.** `handleSubmit` calls `ensureUserProfile({ birthDate })` then **unconditionally** records consent regardless of the result:
```js
const result = await ensureUserProfile({ birthDate, locale });   // line 65
if (userId) {
  await recordConsentBestEffort(
    buildSignUpConsentArgs({ userId, isMinor: isMinorAge, ... }),  // line 70-72
  );
}
```
`isMinorAge` (line 41) is derived from the `birthDate` typed into this screen. But `ensureUserProfile` (auth.ts 442-443) short-circuits when a `public.users` row already exists: `if (existing) return { created: false }` — **without updating `birth_date`**. So when the row pre-exists, the freshly-typed DOB is never persisted, yet `buildSignUpConsentArgs` (consent-selections.ts 65-68) maps `isMinor` straight into the immutable `age_band`/`minor_tier` of a new append-only `consent_records` row. The ledger band can therefore disagree with the stored `birth_date`.

**Trigger.** Reachable when `/complete-profile` renders with `hasProfile` still unresolved (`null`): the guard at line 53 (`if (userId && hasProfile)`) only fires when `hasProfile` is truthy, and line 58 only redirects when `userId === null`; with `userId` set and `hasProfile === null`, the form renders and submit can run. A user re-entering after a profile exists (manual nav, OAuth refresh re-entry before the probe resolves, or — via finding 6 — a transient probe error that mis-routed an established user here) hits `created:false` and appends a consent row keyed on a DOB that was never saved.

**Fix.** When `ensureUserProfile` returns `{created:false}`, skip the fresh consent write (the original sign-up consent already exists), or re-fetch the stored `birth_date` and derive `isMinor` from that before building the args, so the ledger band always matches the persisted DOB.

**Survived-because.** I tried to refute on the `Redirect` guard (line 53) — but it is gated on `hasProfile` being already-true, and AuthContext explicitly initializes/holds `hasProfile: null` during the probe (AuthContext.tsx 132) and even during in-place re-probes (line 120), so the form genuinely renders in the `null` window. I also checked whether `recordConsentBestEffort` is skipped on a duplicate — it is not; it runs on every successful `ensureUserProfile` return, including `created:false`. The ledger is append-only (consent.ts header), so a wrong-band row cannot be corrected, only superseded. Real, though narrow.

### P2 `src/lib/supabase/auth.ts:48-70` [security] — breached-password (HIBP) check fails open on missing Web Crypto, silently removing the only password-strength gate on native

**What / why-real.** `isPasswordBreached` returns `false` (= "not breached, allow") for three distinct conditions: no `crypto.subtle` (line 50-51), non-ok HTTP (line 62), and any thrown exception (line 67-69). `signUpWithEmail` (line 96) only blocks when this returns `true`. On a React Native/Hermes runtime without a SubtleCrypto polyfill, `globalThis.crypto.subtle` is commonly `undefined`, so the check is a **permanent no-op on native** and the only remaining strength enforcement is `password.length >= 8` (sign-up.tsx 63). The file header (auth.ts 30-33) states this is "the code-side backstop, and it matters most for the 14-17 minors," and that the server-side HIBP toggle is a Pro-plan feature the free tier "can't enable" — so there is no server backstop either. A breached password like `password123` (≥8 chars) passes on native.

**Trigger.** Any native (iOS/Android) sign-up on a build without `expo-crypto`/`react-native-quick-crypto` polyfilling `crypto.subtle`. This is the default Hermes posture; the env/config confirms native is a first-class target (the file itself documents the native OAuth path).

**Fix.** On native, wire a SubtleCrypto polyfill so the HIBP range check actually runs; and distinguish "crypto unavailable / HIBP unreachable" (degrade) from "HIBP matched" while logging a warning when the check is skipped, so the gap is observable instead of silent.

**Survived-because.** The strongest refutation is intent: the comment (auth.ts 44-47) explicitly says "Best-effort: if Web Crypto is unavailable ... we DON'T block sign-up." That makes the *fail-open* deliberate — but it does not make the *security gap* non-real. The same file (header 30-33) sells this function as the minor-focused backstop and states the server toggle is unavailable on the free tier, so on native the advertised protection is silently absent exactly where the code claims it matters. "Documented" ≠ "mitigated." I keep this at P2 (not P1): it is a defense-in-depth weakening, not an auth bypass, and length≥8 still applies. Severity is not inflated.
