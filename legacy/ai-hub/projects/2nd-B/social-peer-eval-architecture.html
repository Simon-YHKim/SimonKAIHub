친구/동료평가(peer-eval) — 2nd-B social 정합성 레이어 아키텍처

"타인이 보는 나"를 사용자 확인 게이트로 받아 자기모델 정확도(정합성)를 높이는 구조

2026-06-05 KST · Claude(아키텍처·보안) lane · 병렬: Codex UX 스펙 Grok 소비자 리서치 진행 중 → 종합 예정
왜 이게 "기능 추가"가 아니라 "정확도의 과학적 핵심"인가. deep-research(검증됨): 자기보고는 평가적·도덕적·외현 특성에서 가장 부정확(SOKA, Vazire 2010). 타인 평가가 정확히 그 사각지대를 메우고 증분타당도를 준다(Connelly & Ones 메타 N=44,178). 정확도는 여러 독립 관찰의 누적에서 온다(1~2명도 가치, 5~10명 이상적). 사용자의 "본인 확인 과정"은 *"잠정 제시 + 사용자 검증"* 원칙과 정확히 일치(verdict 방지). → 이 social 레이어는 "정확히 파악하는 구조"의 원칙 #6(외부 관점)을 제품으로 실현한 것.
⚠️ 보안 1순위. 이건 교차사용자 데이터 공유 — 방금 감사가 지적한 knowledge_sources 전역읽기·인젝션과 같은 위험군이다. 설계 철칙: ① 전역 read 절대 금지(친구관계+소유권으로만 스코프) ② peer 텍스트가 LLM(persona/Advisor)에 들어가므로 저장형 프롬프트 인젝션 방어 필수 — 사용자 확인 게이트가 1차 보안 통제(확인된 것만 모델에 진입) + <UNTRUSTED> 펜싱(retrieve.ts 패턴) ③ 미성년(C10) 평가는 별도 보호.

1. 데이터 모델 (제안)

-- 양방향 친구 연결 (동의 기반)
connections(
  id, requester_id → users, addressee_id → users,
  status: 'pending'|'accepted'|'blocked',   -- accepted 만 평가 가능
  -- 무엇을 공유할지 granular consent (양쪽 각각)
  requester_shares jsonb, addressee_shares jsonb,
  created_at, responded_at,
  UNIQUE(requester_id, addressee_id)
)

-- 동료평가 (A가 B에 대해)
peer_evaluations(
  id, evaluator_id → users (A),  subject_id → users (B),
  -- 구조화 신호: 관찰된 강점/특성 (코드화, 자유텍스트와 분리)
  observed_traits jsonb,         -- 예: [{trait, valence, note}] (lexicon-safe)
  free_note_evaluator text,      -- "느낀점 / 내가 보는 너" (UNTRUSTED 취급)
  -- 본인(subject) 확인 상태 — subject 만 변경 가능
  subject_status: 'pending'|'confirmed'|'hidden',
  confirmed_at,
  created_at,
  CONSTRAINT must_be_connected  -- 아래 RLS/CHECK 로 강제
)

provenance(C8 정신): 모든 peer eval은 evaluator + 타임스탬프를 carry. 익명 공격 차단 = 실명·상호 동의 전제(Codex UX/Grok 리서치가 확정).

2. Authz / RLS — 교차사용자 최소권한 (핵심)

작업누가RLS 정책 (개념)
connection INSERTrequester=selfrequester_id = auth.uid()
connection 수락(UPDATE status)addressee=selfaddressee_id = auth.uid() 만 accept/block
peer_eval INSERTevaluator=self + accepted 연결 존재evaluator_id = auth.uid() AND EXISTS(accepted connection between evaluator & subject) — 친구 아닌 사람 평가 불가
peer_eval SELECTsubject 또는 evaluator 본인만subject_id = auth.uid() OR evaluator_id = auth.uid() — 제3자·전역 read 금지
peer_eval 확인/숨김(UPDATE subject_status)subject 만subject_id = auth.uid() — evaluator는 자기 평가의 상태를 못 바꿈
persona 추론 진입시스템subject_status = 'confirmed' 인 것만 자기모델/프롬프트에 사용
인젝션 방어 3중: ① 친구-스코프 INSERT(아무나 못 씀) ② 사용자 확인 게이트(confirmed 만 LLM 진입 — 가장 강한 통제) ③ <UNTRUSTED type="peer_note"> 펜싱 + sanitize(retrieve.ts 재사용). knowledge_sources 감사 교훈을 처음부터 반영.

3. 과학 통합 — peer 신호를 어디에 어떻게 쓰나

1
평가적·외현·도덕 특성에 가중. SOKA상 자기보고가 약한 영역(대인 인상·성실성-실행·도덕)에 peer 신호를 1급으로, 내적 상태(불안·기분)는 자기보고 유지.
2
누적 = confidence. 평가자 수 N을 신뢰도로 표기("3명이 관찰"). 1명도 증분가치, 다수일수록 강함(Connelly & Ones).
3
자기보고 vs 외부관점 분리 표시. persona에 "내가 본 나" / "밖에서 본 나" 두 트랙. 불일치(blind spot)는 *성장 단서*로 잠정 제시 — 단정 금지.
4
삼각측량. 자기보고 + 행동 흔적(기록) + peer 신호 3소스가 수렴하면 고신뢰, 갈리면 "탐구 질문"으로. (Bollich: 일치가 아니라 행동 준거로 벤치마킹.)
5
확인 게이트 = 과학+보안 둘 다. subject가 검토·수용 → 잠정성·자율성 보장(라벨 효과 완화) + 인젝션 차단.

4. 리스크 & 완화

리스크완화
toxicity·harsh 피드백(익명 honest-feedback 앱 셧다운 패턴 — Grok 리서치 확인 중)실명·상호 동의 전제, 강점-우선 구조화, subject가 hide 가능, C9 안전분류기 적용, 금지 lexicon
creepiness·프라이버시("친구가 내 데이터를 봄")opt-in·granular 공유 동의, subject가 무엇이 보이고 쓰이는지 통제, 평가는 "나에 대한 평가"만(데이터 전체 공유 아님)
저장형 프롬프트 인젝션(악의적 친구)§2 3중 방어(친구스코프·확인게이트·UNTRUSTED 펜싱)
관계 손상(부정 평가로 갈등)건설적 프레이밍(Codex), 평가는 subject에게만 보임, 누적·집계로 개별 노출 완화 옵션
미성년(C10)미성년 대상/주체 평가는 별도 정책 — 기본 비활성 또는 보호자 맥락. consent_records 연동

5. Simon 결정 필요 (제품)

  1. 실명 vs 가명 / 상호 공개? (toxicity·신뢰 trade-off — Grok 리서치 반영)
  2. 일방 vs 상호 평가 필수? (상호면 reciprocity, 일방이면 단순)
  3. peer 신호 가중치 — 자기보고 대비 얼마나? (과학은 평가적 특성에 외부 우위라 하나 제품 보정 필요)
  4. 출시 범위 — XPRIZE 데모에 넣을지(차별화 강점) vs v-next(보안·QA 비용)
  5. 프라이버시 기본값 — opt-in 확정(필수)
